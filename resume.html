Application Management Infrastructure upgrade, summer 2001 (Calvin)


Prior to this project, the Student Lab Technical Team (me) had to make a complete partition image of each student lab. Any upgrade to any program required a change to each lab image that used that program, and a full lab re-imaging. After the network server guys got ZENworks working, I was assigned the task of reorganizing the way applications were maintained. Long story short, threw away our old images, made one image per OS (instead of one image per lab), and I used ZENworks (Novell's Application Management System) to create snapshots of each application. ZENworks dynamically installs each program for each lab, and dynamically upgrades any program any time changes were made to the master snapshot for that application.


PractAssist Server, spring/summer 2005 (TAMES)


While my employment at TAMES HRS, Inc. could be summarized as "develop PractAssist", the PractAssist Server deserves special mention. Developing the PractAssist Server gave us the ability to track user data better than our prior processes, create a significantly easier method of data transmission for our clients, and enable us to finally provide in-depth feedback to the users. The PractAssist Server is a Firebird database and a set of Linux daemons that are designed to constantly listen for multiple, simultaneous client programs that wish to request or submit data to us, and facilitate the transfer of that data.

If the client requests any information (including program updates), the network server enters a work order into the database for the request server to deal with. If there is any data that has been submitted with this connection, then the network server also enters a work order for the load server to process. After the request server has signaled that it is finished gathering all the data, the network server gives that data to the client. Splitting the job into three parts (network, request, and load) allows for greater system security and efficiency.

The Firebird database is the heart and soul of the PractAssist Server. There's not much interesting to say about it, though, since it's a database. Funny how that works. OK, I was wrong. The interesting thing is that it formed the basis for the iTAMES website.


iTAMES, 2006-2007


iTAMES is an SSL-secured website that is designed to replace our current method of billing service entry. The secure design of our website required an upgrade in our network infrastructure, which I had to design and implement as part of this project. As project leader, I was responsible for everything, from the original timeline to the thoroughness of the testing phase.

The overall structure of the website is very secure, with each page request being checked with the logged-in user's security level. The data on each page is strictly checked to make sure the data being submitted is valid.

The website is fairly modular. Of course, there is the single-sign-on. But wait! There's more: pages for one type of report are often re-used in other types of reports (function-style, not copy/paste).

Finally, the website is standards-compliant. Viewable in Internet Explorer, Firefox, Opera, and Safari, it uses XHTML 1.0, CSS, and javascript for some polish (unfortunately, tables are still needed, but that's only because IE can't figure out what display:table is).

The website is so well designed that the biggest problem that we have is that our users keep forgetting their passwords. Also, this html resume was based off of my work on iTAMES.

#!/bin/sh
#
# ************************************************************
# anyinstall.sh -- A script to reign in the *nix package
# management madness. Will find your package manager and
# attempt to install the requested package. Intended to be a
# wrapper script with a standard, simplified UI for easy
# installations.
# John Medema, 11/07.
# Freely distributable, no copyrights.
#
# If no switches are given, anyinstall will assume -i.
# Standard Usage (install): anyinstall.sh <package>
# Standard Usage (remove) anyinstall.sh -r <package>
#
# Switches:
# i -- install using the default package manager
# ia -- install using apt-get
# ie -- install using emerge
# ipo -- install using BSD's Ports
# ipa -- install using BSD's Package
# ir -- install using rpm
# h -- print usage info
# r -- remove <package> using the default package manager
# ra -- force remove using apt-get
# re -- force remove using emerge
# rpo -- force remove using BSD's Ports
# rpa -- force remove using BSD's Package
# rr -- force remove using rpm
#
# Switches are not mutually exclusive. `anyinstall.sh -ia -ir <package>`
# will attempt to install <package> first through apt-get, then
# install <package> again via rpm (regardless of the result of
# the attempt with apt-get!).
# Similarly, `anyinstall.sh -rr -ir <package>` will attempt to
# to remove <package> via rpm, then attempt to install <package>
# via rpm.
# As shown by the prior example, SWITCH ORDER IS IMPORTANT!!!!
#
# Notes:
# 1) Supports apt-get, rpm, emerge, and BSD's Ports/Package.
# 2) Will attempt to resolve dependencies on installs.
# 3) Will NOT attempt to remove unneeded dependencies on removes.
# 4) -i acts as install and update. There are no specific
# update commands.
# 5) Only one package per script invocation.
#
# ****************************************************************

# some constants...
usage="anyinstall.sh <package>\nSwitches>: i:   --  install using the default package manager.\nia  --  install using apt-get\nie  --  install using emerge\nipo --  install using BSD's Ports\nipa --  install using BSD's Package\nir  --  install using rpm\nh   --  print usage info\nr   --  remove <package> using the default package manager\nra  --  force remove using apt-get\nre  --  force remove using emerge\nrpo --  force remove using BSD's Ports\nrpa --  force remove using BSD's Package\nrr  --  force remove using rpm\n\n"
apt="apt-get"
aptinst="install"
aptrem="remove"
rpm="rpm"
rpminst="-U"
rpmrem="-e"
emerge=""
emergeinst="emerge"
emergerem="unmerge"
bsdPackage=""
bsdPackageinst="pkg_add"
bsdPackagerem="pkg_delete"
bsdPort=""
bsdPortinst="make build"
bsdPortrem=""


# loop through cmd-line args and determine if we need to search
# for the default package manager.
# Also, we need to find what package to manipulate.
noCmdGiven=1;
useDefault=0;
package="";
for arg in $@;
do
    case $arg in
    -h)
        # assume that the user doesn't know what they're doing and bail.
        echo $usage
        exit
        ;;
    -i|-r)
        useDefault=1
        noCmdGiven=0
        ;;
    -ia|-ie|-ipa|-ipo|-ir|-ra|-re|-rpa|-rpo|-rr)
        noCmdGiven=0
        ;;
    *)
        package=$arg
        ;;
    esac
done


if [ $noCmdGiven -eq 1 ] ; then
    useDefault=1;
fi


# if using the default package manager, then find out what it is by
# checking common locations for the known binaries.
defaultBin=""
if [ $useDefault -eq 1 ] ; then
    # This is the order of our search for our default
    # package manager --> apt-get, rpm, emerge,
    # BSD Package, BSD Port
    for bin in $apt $rpm $emergeinst $bsdPackageinst $bsdPortinst;
    do
        if [ "$defaultBin" != "" ] ; then
            # do nothing
            dummy=""
        elif [ "$defaultinst" != "" ] ; then
            # do nothing
            dummy=""
        else
            for location in `echo $PATH | sed -e 's/:/ /g'`;
            do
                if [ -x $location/$bin ] ; then
                    defaultBin=$location/$bin
                    if [ "$bin" = "$apt" ] ; then
                        defaultinst=$aptinst
                        defaultrem=$aptrem
                    elif [ "$bin" = "$rpm" ] ; then
                        defaultinst=$rpminst
                        defaultrem=$rpmrem
                    elif [ "$bin" = "$emergeinst" ] ; then
                        defaultBin=""
                        defaultinst=$emergeinst
                        defaultrem=$emergerem
                    elif [ "$bin" = "$bsdPackageinst" ] ; then
                        defaultBin=""
                        defaultinst=$bsdPackageinst
                        defaultrem=$bsdPackagerem
                    elif [ "$bin" = "$bsdPortinst" ] ; then
                        defaultBin=""
                        defaultinst=$bsdPortinst
                        defaultrem=$bsdPortrem
                    fi
                fi
            done
        fi
    done
fi





# finally, run through the requested actions.
# but first, a quick hack for when the user just runs with defaults
if [ $noCmdGiven -eq 1 ] ; then
    $defaultBin $defaultinst $package
fi
for arg in $@;
do
    case $arg in
    -i)
        $defaultBin $defaultinst $package
        ;;
    -ia)
        $apt $aptinst $package
        ;;
    -ie)
        $emerge $emergeinst $package
        ;;
    -ipa)
        $bsdPackage $bsdPackageinst $package
        ;;
    -ipo)
        portsdir=$PORTSDIR
        if [ "$portsdir" = "" ] ; then
            if [ -d /usr/ports ] ; then
                portsdir="/usr/ports"
            elif [ -d /usr/pkgsrc ] ; then
                portsdir="/usr/pkgsrc"
            else
                portsdir="/usr/src"
            fi
        fi
        cd $portsdir/$package
        $bsdPort $bsdPortinst
        ;;
    -ir)
        $rpm $rpminst $package
        ;;
    -r)
        $defaultBin $defaultrem $package
        ;;
    -ra)
        $apt $aptrem $package
        ;;
    -re)
        $emerge $emergerem $package
        ;;
    -rpa)
        $bsdPackage $bsdPackagerem $package
        ;;
    -rpo)
        # Ports does not support package removal
        # $bsdPort $bsdPortrem $package
        ;;
    -rr)
        $rpm $rpmrem $package
        ;;
    esac
done
